# Relay Model and Provider Capability Profiles for Execution Routing

## Executive summary

Relay should represent **executor tiers as abstract capability classes**, not as provider-specific prompt templates or hardcoded model names. The current API surfaces from OpenAI, Anthropic, Google Gemini, DeepSeek, Mistral, and Qwen differ materially in context limits, structured-output guarantees, tool-calling behavior, prompt-caching semantics, token accounting, pricing bands, and data-handling defaults. Those differences are large enough that Relay should keep **prompt rendering, token estimation, and cost calculation in provider adapters/config**, while keeping Planner artifacts and canonical packets provider-neutral. ## Relay capability schema and executor tiers

Relay should encode model capability as a **config-driven schema** attached to provider/model records, not embedded into Planner or executor prompts. The canonical packet should carry only abstract routing intent such as ŌĆ£cheap_mechanical,ŌĆØ ŌĆ£mid_implementation,ŌĆØ ŌĆ£strong_semantic,ŌĆØ or ŌĆ£reviewer_auditor,ŌĆØ while a provider adapter resolves that tier into one or more current candidate models and their API-specific flags. That design follows directly from the fact that the providers expose materially different tool semantics, structured-output semantics, caching semantics, and privacy defaults. ## Token and cost accounting

Relay should maintain **provider-specific token accounting**, because the providers are not equivalent here. OpenAI offers an exact token-count endpoint for Responses-format inputs and explicitly counts roles, tools, images, files, and request-structure tokens. Anthropic offers a token-counting API but labels counts as estimates and notes that some system-added tokens are not billed. Google exposes both `countTokens` and per-interaction usage, including multimodal tokenization rules and cached-token accounting. DeepSeek publishes approximate character-to-token ratios and recommends using returned usage values or its tokenizer package. Mistral publishes tokenizer guidance and local tokenizer libraries rather than a comparable first-party count endpoint. ŅłĆciteŅłéturn19view3Ņłéturn6view1Ņłéturn35view0Ņłéturn35view3Ņłéturn22view5Ņłéturn31search0Ņłü

That means Relay should track at least four token measures for every execution candidate: **estimated_input_tokens**, **billed_input_tokens**, **cached_input_tokens or cache_reads**, and **output_tokens**. For some providers, Relay also needs **schema overhead**, **thinking or reasoning tokens where billable**, and **grounding/tool-specific token or request charges**. Google explicitly folds thinking into output pricing for current Gemini Developer API pricing, while Anthropic pricing separates base input, cache-write, cache-read, and output categories. OpenAI adds another wrinkle by using short-context and long-context price bands on some large-context models rather than a single across-the-board rate. ŅłĆciteŅłéturn38search0Ņłéturn36view4Ņłéturn21search2Ņłéturn21search3Ņłü

RelayŌĆÖs cost ledger should therefore support the following billing fields per run:

- `input_uncached_tokens`
- `input_cached_read_tokens`
- `input_cache_write_tokens`
- `output_tokens`
- `thinking_tokens_if_billed`
- `schema_compile_side_effects` as latency metadata, not price unless provider exposes it
- `tool_request_count`
- `grounded_search_queries`
- `cache_storage_token_hours`
- `long_context_band_applied`
- `batch_discount_applied`
- `priority_or_flex_multiplier`
- `region_or_residency_multiplier`. ŅłĆciteŅłéturn21search2Ņłéturn36view4Ņłéturn23view7Ņłéturn22view0Ņłéturn32view1Ņłü

A few provider-specific accounting rules are important enough to encode explicitly. **OpenAI** should be modeled with exact input counting, cached-token details, and optional long-context pricing bands for 1.05M-context models, plus extended prompt-caching state where applicable. **Anthropic** should model base input, 5-minute cache writes, 1-hour cache writes, cache reads, and output as distinct billable categories. **Google** should model raw token price, context-cache storage price, and grounded-search charges separately, and it should persist cached-token usage along with total input/output usage. **DeepSeek** should model cache hit versus miss explicitly on identical-prefix workloads. **Mistral** should model standard input/output plus prompt-cache hits at 10% input-rate equivalent, while allowing local-tokenizer estimates when no provider count endpoint is available. For RelayŌĆÖs actual workflow, the most useful rule is simple: **budget using the providerŌĆÖs native accounting path before dispatch, reconcile using actual usage after execution, and store both values in the audit packet**. That gives operators a ŌĆ£predicted vs actualŌĆØ trail and keeps routing advisory rather than opaque. It also makes cost anomalies visible when a model used more cache writes, more output tokens, or more grounded-search requests than expected. ## Prompt rendering and operator controls

Relay should render the executor brief from the canonical packet into a **provider-neutral intermediate form**, then apply a provider-specific adapter. This is not optional polish. OpenAI, Anthropic, Gemini, DeepSeek, and Mistral each have real behavior differences that affect execution quality and sometimes correctness. ŅłĆciteŅłéturn19view2Ņłéturn6view3Ņłéturn24search4Ņłéturn22view1Ņłéturn32view5Ņłü

For **OpenAI**, Relay should prefer the Responses-style workflow when possible, because OpenAI explicitly recommends Responses for reasoning workloads and provides stateful conversation management, structured outputs, and prompt-caching guidance around that path. Stable, reusable content should be placed early in the request to maximize cache hits. Relay should also keep tool descriptions outside the human-readable executor brief when they can be carried as structured tool definitions. ŅłĆciteŅłéturn21search11Ņłéturn37search6Ņłéturn37search3Ņłéturn37search7Ņłü

For **Anthropic**, Relay should assume tool definitions materially affect model behavior. Anthropic constructs a special system prompt from tool definitions and tool configuration, and its docs explicitly recommend long, detailed tool descriptions. That means Anthropic prompt rendering should emphasize concise executor instructions in human text, while pushing interface contract detail into JSON Schema and tool descriptions. Relay should also recognize that structured outputs and strict tool use may add a first-call latency hit due to grammar compilation and that schemas are cached for up to 24 hours. ŅłĆciteŅłéturn6view3Ņłéturn34view3Ņłéturn34view4Ņłéturn34view5Ņłü

For **Google Gemini**, the most important routing implication is whether Relay uses **stateful Interactions** or manually managed stateless history. In stateful mode, the server manages history and thought signatures. In stateless mode, Relay must preserve thought blocks and signatures exactly, especially in function-calling loops, or the request can fail validation. That is a concrete reason for RelayŌĆÖs canonical packet to stay provider-neutral while the adapter layer decides how state is serialized. ŅłĆciteŅłéturn24search4Ņłéturn24search5Ņłéturn35view3Ņłü

For **DeepSeek**, Relay should render explicit JSON instructions whenever `json_object` is requested, because DeepSeek warns that otherwise the model may emit an endless whitespace stream until token limit exhaustion. Relay should also know that strict function calling is a beta-only mode and that DeepSeek exposes both OpenAI-compatible and Anthropic-compatible APIs. In practice, that makes DeepSeek attractive as a price-sensitive route, but only if the adapter owns the quirks. ŅłĆciteŅłéturn22view2Ņłéturn22view1Ņłéturn22view0Ņłü

For **Mistral**, Relay should explicitly specify JSON or JSON Schema response formats and add a stable `prompt_cache_key` for repeatable coding sessions. Mistral also enables parallel tool calls by default, which can be an advantage for reviewer or repo-inspection workflows but should remain operator-controllable in Relay. ŅłĆciteŅłéturn12search0Ņłéturn32view0Ņłéturn32view1Ņłéturn13search7Ņłü

The operator override model should mirror RelayŌĆÖs product principles. A good v1 pattern is:

- **Policy allowlist**: workspace or repo defines which providers, regions, preview models, and built-in tools are allowed.
- **Advisory recommendation**: Relay computes candidate tiers and provider/model options with cost, context, privacy, and capability notes.
- **Operator selection**: a human chooses the actual provider/model, reasoning level, context budget, and redaction mode before dispatch.
- **Audit trail**: Relay stores what was recommended, what was chosen, and why the operator overrode it.  

That preserves manual control, keeps routing explainable, and fits the fact that benchmark standing, preview availability, and cost curves are moving quickly across all major providers. ŅłĆciteŅłéturn26view0Ņłéturn26view1Ņłéturn33view0Ņłéturn24search8Ņłü

## Risks, privacy, and benchmark caveats

The biggest practical risk is **false uniformity**. OpenAIŌĆÖs extended prompt caching, AnthropicŌĆÖs grammar-cached structured outputs, GeminiŌĆÖs thought signatures, DeepSeekŌĆÖs beta-only strict tool calling, and MistralŌĆÖs `prompt_cache_key` are not interchangeable features with different names; they are meaningfully different operational contracts. If Relay normalizes them too aggressively, it will produce bad cost estimates or brittle executor behavior. ŅłĆciteŅłéturn37search0Ņłéturn34view4Ņłéturn24search5Ņłéturn22view1Ņłéturn32view1Ņłü

The privacy and retention picture is also different enough that Relay should keep **provider-specific redaction policy** in config. OpenAI says API data is not used to train models by default and documents application-state retention categories including prompt caching and other stateful features. Anthropic says commercial API inputs and outputs are not used for training by default, deletes API inputs/outputs within 30 days absent exceptions, and offers ZDR on many but not all features. GoogleŌĆÖs free Gemini API tier is used to improve Google products, while paid tiers are not, and Google also supports opt-in logging/sharing flows. Mistral says API data is not used for model training and offers privacy controls and ZDR distinctions. DeepSeekŌĆÖs public documentation is less explicit about a commercial-API non-training default, while its broader privacy pages do describe collection and use of user input to improve technology; that should trigger additional caution for proprietary repositories. ŅłĆciteŅłéturn19view4Ņłéturn17search14Ņłéturn18search0Ņłéturn18search11Ņłéturn18search1Ņłéturn16search1Ņłéturn23view7Ņłéturn23view6Ņłéturn32view2Ņłéturn28search9Ņłéturn9search0Ņłéturn9search6Ņłü

Benchmarks are useful, but only in a **layered** way. SWE-bench Verified is valuable because it uses real GitHub issues and execution-based scoring on a human-filtered subset. LiveCodeBench is valuable because it continuously harvests fresh coding problems and measures code generation, self-repair, and execution. AiderŌĆÖs leaderboard is useful as a practitioner benchmark for code editing and edit reliability. But none of these is a perfect proxy for RelayŌĆÖs bounded handoff pipeline, because harness design, tool scaffolding, retries, allowed context, and model-specific adapters all materially affect scores. Relay should therefore store benchmark signal as advisory metadata with source, date, and harness notes instead of turning it into deterministic routing logic. ŅłĆciteŅłéturn26view2Ņłéturn26view3Ņłéturn26view1Ņłéturn26view4Ņłü
