# Relay MCP context packet schema drift root-cause note

## 1. Confirmed runtime DB path

The MCP server selects its SQLite path from `RELAY_DB_PATH`; if that variable is empty, it uses `data/relay.sqlite`. The startup log includes the selected `db_path`.

In this workspace, `.env`, `.env.local`, and the current shell environment did not define `RELAY_DB_PATH`, so the MCP default is `data/relay.sqlite`. I inspected that DB. It also contains project `1`, which matches the reported working MCP calls against project `1` / repo `Relay`.

External MCP client startup logs were not available from this diagnostic run. If the client launches MCP with its own `RELAY_DB_PATH`, that client log/config remains the final confirmation point. The evidence below is from the workspace default MCP DB.

## 2. Actual runtime context_packets schema

`PRAGMA table_info(context_packets);` against `data/relay.sqlite` shows this table shape:

Confirmed absent from the runtime table:

- `project_row_id`
- `source_snapshot_row_id`
- `summary_json`
- `completed_at`

`PRAGMA table_info(context_packet_sources);` shows the expected source rows table, including `context_packet_row_id`, `source_id`, `source_type`, `project_id`, `repo_id`, `source_snapshot_id`, path/range/hash/redaction fields, and timestamps.

`PRAGMA table_info(source_snapshots);` shows `source_snapshots.project_row_id` exists in the same DB. That localizes the drift to `context_packets`; row-id-backed source snapshots are not generally broken.

## 3. Goose migration state in that DB

`SELECT * FROM goose_db_version ORDER BY id DESC;` against the same DB shows version `10` is marked applied:

```text
id  version_id  is_applied  tstamp
11  10          1           2026-06-23 02:05:05
```

Later versions are also applied through version `12`. This confirms the issue is not solved by rerunning ordinary migrations: goose already considers migration 10 applied in this DB.

The context packet count in this DB is currently `0`, so no existing context packet rows need data preservation in this specific inspected DB. The durable repair should still preserve rows because other operator DBs may already contain context packet data.

## 4. Source-controlled expected schema

Current `internal/db/migrations/00010_context_packets.sql` creates `context_packets` with:

- `project_row_id INTEGER NOT NULL REFERENCES projects(id) ON DELETE CASCADE`
- `source_snapshot_row_id INTEGER NOT NULL REFERENCES source_snapshots(id) ON DELETE CASCADE`
- `summary_json TEXT NOT NULL DEFAULT '{}'`
- `completed_at TEXT NOT NULL DEFAULT ''`

Current `internal/db/queries/context_packets.sql` inserts and selects `project_row_id`, `source_snapshot_row_id`, `summary_json`, and `completed_at`.

Current generated sqlc code in `internal/store/generated/context_packets.sql.go` uses explicit column lists that reference those same columns for `CreateContextPacket`, `GetContextPacketByID`, and `ListContextPacketsByProject`.

The store wrapper and service pass these fields through:

- `internal/store/context_packets.go` includes `ProjectRowID`, `SourceSnapshotRowID`, `SummaryJSON`, and `CompletedAt`.
- `internal/contextpackets/service.go` resolves the project row ID and source snapshot row ID, then passes them into `CreateContextPacket`.

## 5. Root cause classification

Confirmed classification for the inspected default MCP DB: stale applied migration / runtime schema drift.

The runtime `context_packets` table has an older shape, while `goose_db_version` marks migration 10 applied. That combination is incompatible with the current source-controlled migration 10 and generated sqlc code. The strongest explanation is that `00010_context_packets.sql` changed after this DB had already recorded migration 10 as applied, so goose will not replay it and the table remains in the old shape.

Wrong-DB selection is not supported by the local workspace evidence: both `cmd/mcpserver` and `cmd/relay` default to `data/relay.sqlite`, no local env override was found, and the inspected DB contains project `1`. However, an externally configured MCP client can still override `RELAY_DB_PATH`; check its startup `db_path` log line before applying a repair to a different database.

## 6. Explanation of failing MCP paths

`create_context_packet` fails because `HandleCreateContextPacket` calls `contextpackets.NewService(...).CreateContextPacket(...)`, which eventually calls the generated `CreateContextPacket` SQL. That SQL inserts `project_row_id`; SQLite rejects the statement because the runtime table has no such column.

`get_pass_context` fails while loading latest context packet metadata because `loadLatestContextPacketMetadata` calls `ListContextPacketsByProject`. The generated query explicitly selects `project_row_id`, `source_snapshot_row_id`, `summary_json`, and `completed_at`; SQLite cannot execute that query against the drifted table.

`get_context_packet(ctx_not_real, include_sources=true)` returns `INTERNAL_ERROR` instead of clean `NOT_FOUND` because `HandleGetContextPacket` can only map `sql.ErrNoRows` after `GetContextPacketByID` executes successfully. With the drifted schema, the generated SELECT fails on a missing column before a no-row result can be produced.

## 7. Alternate bootstrap/test path findings

Repo search found only `internal/db/migrations/00010_context_packets.sql` as a real `CREATE TABLE context_packets` source.

No alternate bootstrap schema, fixture SQL, copied prebuilt SQLite DB, or script-created `context_packets` schema was found in `cmd`, `internal`, `scripts`, docs, or tests. Test code opens temporary SQLite DBs through `store.Open`, which runs the embedded goose migrations.

## 8. Durable fix recommendation

Do not edit `00010_context_packets.sql` in place, and do not rely on rerunning migration 10. The DB has already recorded version 10 as applied.

Use a new forward-only migration that repairs drifted DBs safely:

- Detect whether `context_packets.project_row_id` and the other current columns are missing.
- Rebuild `context_packets` into the current schema, or add nullable columns, backfill, then rebuild into the final NOT NULL schema.
- Backfill `project_row_id` by joining `context_packets.project_id` to `projects.project_id`.
- Backfill `source_snapshot_row_id` by joining `context_packets.source_snapshot_id` to `source_snapshots.source_snapshot_id`.
- Populate `summary_json` with `'{}'` and `completed_at` with `''` for legacy rows.
- Preserve existing `context_packets` rows.
- Preserve existing `context_packet_sources` rows and their `context_packet_row_id` references.
- Recreate the existing context packet indexes.
- Fail the migration if any row cannot be backfilled to a valid project/source snapshot row.

Add regression coverage that starts from the stale table shape, marks migration 10 applied, then verifies the new repair migration produces the current schema and allows the generated context packet queries to run. Also add a schema expectation check for generated sqlc/runtime DB alignment so this class of drift fails earlier.
